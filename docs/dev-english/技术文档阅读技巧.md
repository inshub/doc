# 技术文档阅读技巧：我踩过的那些坑

哈喽，大家好，我是纯白。


刚毕业那会儿，我特别崇拜能把文档从头读到尾的人。

觉得他们厉害，自律，扎实。所以我试着这样干——打开一个技术文档，从第一页开始，一行一行往下读。

结果呢？读了十页就困了。二十页之后，什么都不记得。

后来我才明白，技术文档不是小说，不能那样读。


今天说几个我这些年总结的技巧。

不是那种"教你读书"的大道理，是我真的踩过坑、试过错之后，觉得有用的东西。


## 先读目录，别急着翻正文

这个我吃了亏。

以前拿到文档就直奔第一章，觉得"先按顺序来嘛"。读了半天才发现，前面几十页都是介绍背景、历史、哲学思想的，我真正需要的东西在第三章。

后来我学乖了。

拿到文档先看目录，花两分钟扫一遍：
- 这文档大概讲什么
- 哪些是我需要的
- 哪些可以跳过

然后直接跳到需要的章节。

前两天看一个前端框架的文档，目录有十五章。我扫了一遍，发现只有第三章、第七章、第十一章是我需要的。其他的不是"最佳实践"就是"架构设计"，跟我当前的任务没关系。

我就没管那十二章，直接看了三章，搞定。

有人可能会说，前面那些背景知识不重要吗？

实话实说，很多时候不重要。或者说，不重要到可以以后再说。

你需要什么，就去读什么。等有空了，再回来补背景。

技术文档不是考试教材，不用按顺序读。


## 带着问题去读，别当闲书看

这个我之前也做得不好。

拿起文档就开始读，读着读着就开始走神。一会儿觉得这段写得挺深奥，一会儿觉得那段说得有道理，读完了什么也没记住。

后来我发现，得带着问题去读。

比如我要用这个库实现一个上传功能。那我打开文档之前，先问自己：
- 我要解决什么问题？
- 我需要知道哪些信息？
- 有哪些关键词是我要找的？

然后读的时候，只看跟问题相关的内容。其他的一概略过。

这跟查字典似的，你不会从"A"开始查到"Z"，你是知道要查什么字，直接翻到那一页。

技术文档也是一样的。

我见过太多人，打开文档就开始从头读，读了半天觉得"学到了很多"，但问他"你会用了吗"，他说"还不会，得再看看"。

这就是没有带着问题读。

你先得知道自己要解决什么，然后去文档里找答案。

文档不是用来"学习"的，是用来"查"的。


## 边读边敲，别光用眼看

这个太重要了。

我以前读文档，就是眼睛看。看完觉得"懂了"、"会了"。结果一上手，发现根本不是那么回事。

后来我学乖了——读文档的时候，终端开着。

读到一段，就跟着敲一段。把代码跑一遍，看看输出是什么，跟文档说的是不是一样。

很多时候，文档写得再好，不如你跑一遍代码来得实在。

上次学一个新工具，文档说"这个参数默认是 false"。我没信，自己写了个小 demo 测试了一下，发现默认值其实是 null。

不是说文档错了，可能是版本变了，也可能是我理解错了。但这都不重要，重要的是你自己验证过。

技术这东西，看十遍不如自己敲一遍。

我现在读文档，基本就是：
1. 打开终端
2. 读一段
3. 敲一段
4. 跑一遍
5. 继续读

慢是慢了点，但读完之后，真的会用了。


## 别追求100%理解

这个我之前也纠结过。

总觉得文档得"彻底读懂"、"全部掌握"，不然心里不踏实。

结果就是在一个地方卡住，死磕，浪费了很多时间。

后来我明白了一个道理：够用就行。

你不是要成为这个技术的作者，你是要用它来解决实际问题。你能用起来，就够了。

至于那些深层的原理、边缘的场景、优化的技巧，以后再说。

我见过很多人（包括以前的我），在文档的一个小细节上卡几个小时，最后发现那个细节根本用不上。

别追求完美。

能用，就够了。


写这篇文章的时候，我翻了翻之前读文档时做的笔记。

有些笔记已经看不懂了，有些笔记现在看起来很幼稚。

但那些真的动手敲过的代码、真的踩过的坑，现在还记得。

技术文档这东西，不是读出来的，是用出来的。

先从一个小问题开始，带着问题去查，边查边试，用起来再说。

别想着"我先把文档读完再开始做"，那样你可能永远也读不完。
